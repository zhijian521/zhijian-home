# 知简重构迁移审查总表

> 本文档是 `zhijian` 向 `zhijian-home` 迁移期间的唯一进度基线。
> 迁移不是复制旧代码，而是逐项确认需求、审查旧实现、决定保留或重写、完成新实现，并由人工审查通过。

## 1. 项目与目标

| 项目 | 路径 | 角色 |
| --- | --- | --- |
| 旧项目 | `C:\code\Zhijian Code\zhijian` | 功能和行为参考，不视为可信实现 |
| 新项目 | `C:\code\Zhijian Code\zhijian-home` | 重构迁移后的正式项目 |

本次迁移有三个同时进行的目标：

1. **迁移**：保留经过确认的产品能力和必要数据。
2. **重构**：重新建立清晰的目录、依赖边界、类型和样式体系。
3. **审查**：逐项检查旧代码的正确性、安全性、性能、可访问性和可维护性，不默认旧实现正确。

## 2. 强制执行规则

### 2.1 每次只迁移一个可审查单元

一个迁移单元应满足：

- 通常只覆盖一个页面、一个用户流程，或一组不可拆分的基础设施。
- 相关改动可以独立构建、验证和回退。
- 提交前能够用几分钟说明清楚旧行为、发现的问题和新实现。
- 不把多个互不相关的页面或 API 混在同一次迁移中。

### 2.2 “已实现”不等于“已通过”

进度必须同时记录两个状态：

| 字段 | 可选值 | 含义 |
| --- | --- | --- |
| 实现状态 | `未开始` / `分析中` / `实现中` / `已实现` / `不迁移` | 代码工作进度 |
| 人工审查 | `未审查` / `审查中` / `需修改` / `已通过` / `不适用` | 用户的审核结论 |

只有同时满足以下条件，迁移单元才算真正完成：

- 实现状态为 `已实现` 或经确认的 `不迁移`；
- 人工审查状态为 `已通过`；
- 自动检查通过；
- 本文档已记录结论和对应提交。

AI 完成代码后只能把条目标记为 `已实现 / 未审查`，不得代替用户标记 `已通过`。

### 2.3 不直接复制未经审查的旧代码

每个迁移单元开始前必须先回答：

1. 旧功能实际解决什么问题？
2. 该功能是否仍然需要？
3. 旧实现有哪些正确性、安全、性能、体验或结构问题？
4. 哪些内容可以保留，哪些需要重写，哪些应删除？
5. 新实现的验收标准是什么？

发现旧代码问题时，先把问题和处理决定记录到本文件或该单元的审查记录，再写新实现。

### 2.4 Git 与提交边界

- 原则上一个迁移单元对应一个 Conventional Commit；确需拆分时，提交仍须围绕同一单元。
- 不在迁移提交中顺手修改无关模块。
- 每个条目完成实现后记录提交哈希。
- 人工审查要求修改时，在同一条目的“审查记录”中追加结论，不覆盖历史结论。

## 7. 每次人工审查应看到的内容

提交一个条目给用户审查时，说明必须控制在以下范围内：

1. **本次迁移了什么**：只描述本单元。
2. **旧实现发现了什么问题**：按严重程度列出，不粉饰旧代码。
3. **做了什么决定**：保留、重写、删除和延后的内容。
4. **如何验证**：命令结果、页面路径、截图或关键场景。
5. **需要用户判断什么**：视觉、交互、产品取舍或风险接受。
6. **明确当前状态**：在用户确认前保持 `已实现 / 未审查`。

## 8. 通用审查清单

并非每个单元都需要所有项目，但相关项不得跳过。

### 8.1 产品与行为

- 功能是否仍然有价值，是否存在更简单的实现？
- 正常、空数据、失败、加载和无权限状态是否完整？
- 是否保留了旧项目偶然形成、但没有产品价值的行为？
- 页面和 API 的边界是否清晰？

### 8.2 正确性与类型

- 是否存在边界条件、日期/时区、分页或并发问题？
- 不可信输入是否先校验，再进入业务逻辑？
- 是否滥用 `any`、类型断言或非空断言？
- 错误是否保留足够上下文，同时不向客户端泄漏敏感信息？

### 8.3 安全

- 鉴权是否发生在可信的服务端边界？
- 是否存在越权、XSS、CSRF、SSRF、SQL 注入或路径穿越风险？
- Cookie、Token、密钥和用户数据的处理是否安全？
- 上传、外部 URL 和开放写入接口是否有限制、超时和限流？

### 8.4 性能

- 是否出现重复请求、串行等待、无界查询或不必要的客户端组件？
- 图片、Markdown、图表和大列表是否有合理加载策略？
- 缓存是否会造成错误数据或权限数据串用？
- 数据库查询是否有明确分页和索引依据？

### 8.5 可访问性与视觉

- 页面结构、标题、表单标签和按钮语义是否正确？
- 键盘焦点、弹窗焦点管理、颜色对比度是否可用？
- 动画是否尊重 `prefers-reduced-motion`？
- 桌面、移动端、亮色和暗色模式是否都经过检查？

### 8.6 可维护性

- 依赖方向是否保持 `app -> domain -> core`，且职责清晰？
- 是否复制了本应删除的旧抽象或重复逻辑？
- 组件和函数是否足够小，但没有为了抽象而抽象？
- 文档、配置、Schema 和实际实现是否同步？
