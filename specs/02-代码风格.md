# 02 · 代码风格

## 总则

代码读起来要像同一个人写的。简洁优先,注释讲清「为什么」而非「是什么」。

## 文件头与分区注释

沿用原项目的注释块风格。文件顶部用 `/*== ... ==*/` 框出文件用途;文件内用 `/*== 分区 ==*/` 分段:

```ts
/*============================================================================
  api-response — 统一 API 响应结构

  所有 API Route 返回此格式:
  成功 { code: 0, data, message }
  失败 { code: xxx, data: null, message }
============================================================================*/

/*== 业务错误码常量。 ==*/
export const BizCode = { ... } as const;
```

- 顶部块:一句话说文件做什么 + 关键约定。
- 分区块:文件超过 ~40 行就分段,每段一个 `/*== xxx ==*/` 标题。
- 单行小区块用 `/*== xxx ==*/` 紧贴代码上方。

## 命名

| 对象 | 规则 | 示例 |
|---|---|---|
| 文件 / 目录 | kebab-case | `api-response.ts` `ghost-button/` |
| 组件文件 | kebab-case,导出 PascalCase | `ghost-button.tsx` → `export function GhostButton` |
| 函数 / 变量 | camelCase | `requireAdminFromRequest` |
| 类型 / 接口 | PascalCase | `SessionPayload` `ApiResponse` |
| 常量对象 | PascalCase(作命名空间) | `BizCode` |
| CSS 变量 | `--kebab-case` | `--space-4` `--text-md` |
| CSS class | camelCase(CSS Modules) | `.submitButton` |

## TypeScript

- 严格模式,`noImplicitAny` 开。
- 公共函数显式标注返回类型。
- 禁止 `any`。信任边界(解析外部 JSON、第三方返回)需 `any` 时,旁边注释说明为什么。
- 类型就近定义,不建 `types/` 大杂烩文件。`posts.ts` 的类型就在 `posts.ts` 里。

## 导入顺序

```ts
// 1. node 内置 / next
import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';

// 2. 项目内(@/ 别名)
import { requireAdminFromRequest } from '@/lib/core/auth';
import { BizCode, fail } from '@/lib/core/api-response';

// 3. 同目录相对
import { listSomething } from './queries';
```

每组之间空一行。type import 用 `import type`。

## 工具链

格式由 **ESLint + Prettier** 保证,配置见项目根:

| 文件 | 作用 |
|---|---|
| `.prettierrc.json` | 4 空格 / 160 字符换行 / 单引号 / 尾逗号 |
| `.editorconfig` | 缩进/换行兜底,任何编辑器都认 |
| `eslint.config.mjs` | Next 预设 + 关 Prettier 冲突规则 |

命令:

```bash
npm run format        # prettier --write .
npm run format:check  # 校验不改
npm run lint          # eslint .
npm run typecheck     # tsc --noEmit
```

核心约定:**4 空格缩进、单行不超 160 字符、单引号、分号、尾逗号**。提交前三工具必绿。

## 简洁要求

- **消灭样板**:同样的 403 返回写三遍就该包成 `withAdmin`。先看有没有,再写。
- **不抽象只有一个实现的接口**。一个 product 不配 factory。
- **删除优先**:能删的代码不注释,注释掉的死代码直接删。
- **短函数**:一个函数做一件事。超过屏幕要滚,考虑拆。
- **刻意简化要标注**:`// ponytail: 全局锁,若按账号有吞吐需求再换 per-account 锁` —— 说清上限与升级路径。

## 留自检

非平凡逻辑(分支/循环/解析/金钱/鉴权)留一个最小可跑的自检:文件末尾 `__main__` 风格的 `demo()` 或一个 `test_*.ts`。纯一行的不需要。

## 禁止

- 禁 console.log 提交(调试用,删掉)。
- 禁 TODO 不带人名/上下文(`// TODO(ponytail): 待 DB 字段确认`)。
- 禁「为以后」的空文件、空目录、空接口。
