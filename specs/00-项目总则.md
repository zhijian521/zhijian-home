# 00 · 项目总则

## 目标

「知简」是一个个人博客 + 导航站 + 后台管理的 Next.js 站点。v2 的目标不是简单搬运，而是把旧仓业务按新的结构和约束重写出来。

核心目标只有三个：

- 简洁：能复用就不重写，能删掉就不保留，能一层解决就不绕两层。
- 整齐：同类东西放在同类位置，命名、分层、依赖方向稳定一致。
- 完整：功能闭环，接口有文档，组件可预览，规则有校验兜底。

## 技术栈

| 项 | 选型 | 备注 |
|---|---|---|
| 框架 | Next.js 15 App Router | 不使用 Pages Router |
| 语言 | TypeScript 5 | 严格模式 |
| 运行时 | Node.js | API Route 跑在服务端 |
| 数据库 | MySQL + `mysql2/promise` | 连接池单例 |
| 样式 | CSS Modules + 原生 CSS 变量 | 不引入 Tailwind / CSS-in-JS |
| Markdown | `react-markdown` + `remark-gfm` + `rehype-highlight` | 博客与后台文档共用 |
| 图表 | `recharts` | 仅后台分析使用 |

新增依赖前，先按这个顺序自问：

1. 现有实现能不能复用？
2. Node/浏览器原生能力能不能解决？
3. 已安装依赖能不能解决？
4. 如果都不行，再引新依赖，而且必须能说明收益。

## YAGNI 优先级

写任何东西前，按这个顺序判断，停在第一个成立的位置：

1. 这东西现在真的需要吗？
2. 项目里已经有了吗？
3. 标准库能做吗？
4. 平台原生能做吗？
5. 已装依赖能做吗？
6. 能不能用最小实现做完？

不要为了“以后可能用到”提前建接口、目录、抽象层、工厂或空文档。

## 重构边界

- v2 从零重写，不做旧仓那种 `re-export` 过渡层。
- 旧仓提供业务规则、SQL 结构、接口语义、交互细节。
- v2 重写的是目录结构、模块边界、规范体系和代码组织方式。
- 迁移时优先保留行为一致性，不为了“更优雅”改掉原有产品逻辑。

## 明确不做

- 不做全自动反射式 showcase / API 扫描。
- 不做全局 `modules/` 空壳 CSS。
- 不做只有一个实现的接口抽象。
- 不做没有落地校验的“建议型规范”。

## 强制校验

提交前统一运行：

```bash
npm run check
```

当前 `check` 至少包含：

- `npm run lint`
- `npm run typecheck`
- `npm run docs:check`

如果只是大范围整理格式，可额外运行：

```bash
npm run format
```

但“这次改动是否完成”的判断，以 `npm run check` 为准。
