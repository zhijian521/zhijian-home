# 07 · 文档与校验

## 两类文档

| 类型 | 位置 | 形式 |
|---|---|---|
| 功能文档 | `src/docs/features/*.md` | Markdown |
| 接口文档 | `src/docs/api/_registry.ts` | 手写 registry + route 顶部注释 |

文档不是附属物，而是迁移过程的一部分。改功能时，文档必须跟着改。

## 功能文档

功能文档放在 `src/docs/features/`，由 `src/docs/features/_registry.ts` 统一登记。

每篇文档至少要说明：

- 这个模块做什么
- 页面 / 组件 / 数据流入口在哪里
- 特殊约束或容易踩坑的点

## 接口文档

接口文档由两部分组成：

1. `route.ts` 顶部 JSDoc
2. `src/docs/api/_registry.ts`

二者缺一不可。registry 负责给后台文档中心渲染索引，JSDoc 负责把信息留在代码旁边。

## `/admin/docs`

后台文档中心最终承载两块内容：

- 功能文档 tab：读 `DOC_REGISTRY`
- 接口文档 tab：读 `API_REGISTRY`

它和 `/admin/showcase` 一样，属于后台基础设施页面。

## docs:check 负责什么

`scripts/docs-check.mjs` + `npm run docs:check` 现在应负责下面几件事：

1. `DOC_REGISTRY` 里登记的功能文档文件都存在
2. 磁盘上的 `src/docs/features/*.md` 都已登记到 `DOC_REGISTRY`
3. 如果仓库里已经出现任何 `src/app/api/**/route.ts`，则 `src/docs/api/_registry.ts` 必须存在
4. 如果 `API_REGISTRY` 存在，则其中每个 path 都必须能映射到真实 route 文件

以后可以继续增强，但不要一开始就做复杂解析器。

## 手写 registry，不做反射

坚持手写 registry 的原因：

- Next 的动态路由和条件导出不适合搞脆弱的自动扫描
- 文档组织本来就需要人工判断
- 手写 + 校验，维护成本低，结果也更稳定

## 提交流程

提交前统一运行：

```bash
npm run check
```

它是当前的统一完成标准。单独跑 `docs:check` 只适合在改文档/registry 时做快速验证，不替代最终检查。
