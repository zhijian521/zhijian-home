# 07 · 文档与校验

## 两类文档

| 类 | 位置 | 形式 |
|---|---|---|
| 功能文档 | `src/docs/features/*.md` | Markdown,react-markdown 渲染 |
| 接口文档 | `src/docs/api/_registry.ts` | 手写 registry,后台渲染 |

文档与代码同目录树——改代码时文档在眼前,顺手改。

## 功能文档

- 每个大功能一篇 `.md`:文章管理、导航站、统计、AI 对话、上传管理……
- 文件放 `src/docs/features/`,命名 `posts.md` `nav.md`。
- 索引 `src/docs/features/_registry.ts`(`DOC_REGISTRY`):登记每个 `.md` 的路径 + 标题 + 所属模块。

## 接口文档

见 [04-API规范](./04-API规范.md) 的注释规范与 `_registry.ts`。route 顶部 JSDoc + registry 手写登记,两者一致由校验兜底。

## 后台文档中心

`/admin/docs` 双 tab,复用 admin 鉴权:

- **功能文档 tab**:渲染 `DOC_REGISTRY`,左侧树形目录(按模块),右侧 react-markdown。
- **接口文档 tab**:渲染 `API_REGISTRY`,按 group 分组列表,每条显示路径/鉴权/方法/说明。

另有 `/admin/showcase` 组件/图标预览(见 [06](./06-组件规范.md))。三个后台页面共享 admin 鉴权与布局。

## docs:check 校验

`scripts/docs-check.mjs` + `npm run docs:check`,校验三件事:

1. `API_REGISTRY` 列的每个接口路径 → 对应 `app/api/**/route.ts` 文件存在。
2. `DOC_REGISTRY` 列的每个 `.md` → 文件存在。
3. 磁盘上的 `docs/features/*.md` → 都在 registry 登记(防漏登)。

可选增强(后续加):
4. route 顶部 JSDoc 的 `@api path` 与 registry 的 path 一致。

任一不通过,退出码非 0。提交前必跑。

## 手写 registry,不做反射

接口与组件都手写登记。原因:

- Next 下全自动反射(动态路由、条件导出)fragile。
- 示例 props 由开发者定,可控。
- 注释在代码旁,改接口顺手改注释,再顺手改 registry——一次性投入,后续维护成本低。

代价是 33 个文件加注释 + 2 个 registry 手写。可接受。

## 提交流程

```bash
npm run typecheck   # 必过
npm run docs:check  # 必过
# 两个都绿才算这次改动完成
```

后续接 git pre-commit hook 自动跑,过不了不让提交。
