## 三条总要求

1. 代码要简洁，避免样板和为未来预留的空壳。
2. 结构要清楚，目录职责、依赖方向、组件层级都要稳定。
3. 规范要可执行，文档不能只写原则，必须能落到脚本或清单上。

## 冲突时怎么判定

按下面的优先级处理，越靠前越优先：

1. 仓库内可执行事实：脚本、配置、已提交实现
2. 更具体、更贴近当前主题的专项规范
3. 更泛的总则、索引说明

也就是说，如果某条 spec 和 `package.json`、校验脚本、真实变量名短暂漂移，先以可执行事实为准，并尽快把文档修回一致。

## 与原项目的关系

旧仓 `zhijian` 主要提供业务逻辑、数据结构和历史方案参考。`zhijian-home` 是 v2 的重写仓，结构、约束、校验都以本目录为准，不沿用旧仓里的渐进式过渡策略。

## 统一完成标准

日常开发统一以 `npm run check` 作为完成前校验入口。当前它应覆盖：

- `npm run lint`
- `npm run typecheck`
- `npm run docs:check`

格式化由 `npm run format` / `npm run format:check` 辅助，但提交门禁以 `check` 为准，除非后续显式把格式校验加入 `check`。
